## 1. Storage Structure

```text
knowledge_base/               # git repo root (GitHub private remote)
  .git/                       # NOT synced to GDrive (rclone filter)
  INDEX.md                    # Auto-generated: yearly links + topic MOC
  2026-03/
    INDEX.md                  # Auto-generated: monthly file list with summaries
    20260312-{slug}.md
    20260315-{slug}.md
  moc/
    {tag}-MOC.md              # Auto-generated: per-tag file list, latest-first
  scripts/
    generate_indexes.ts       # Regenerates all INDEX.md and moc/*.md from frontmatter
    sync_knowledge.sh         # git add/commit/pull --rebase/push with error handling
```

> `INDEX.md` and `moc/` files are **always generated by script**, never hand-edited by agent.

### File Format

```yaml
---
title: NanoClaw와 FastAPI 포트 충돌 해결
created_at: 2026-03-12T14:30:00Z
tags: [nanoclaw, fastapi, bugfix]
---
# 요약
...
[[20260315-director-pattern]] 참고
```

> `created_at` is written by the agent at file creation time (ISO timestamp). Human-readable only.
> Authoritative history is always in git (`git log`). Files can be freely overwritten — git retains all previous versions.

### Policies

- **Overwrite freely**: no grace period enforcement. git history is the safety net for all previous versions.
- **Tags only**: no category field. Classification via tags only.
- **Wikilinks**: `[[slug]]` format, slug = filename without extension (e.g., `[[20260312-nanoclaw-port]]`). `ReadNoteTool` resolves via `glob("**/{slug}.md")`.
- **INDEX/MOC**: always auto-generated by `generate_indexes.ts`. Never manually edited.

### GDrive Sync (rclone filter)

File `~/.config/rclone/knowledge_base.filter`:

```text
- .git/**
- scripts/**
- *~
- .DS_Store
+ **
```

## 3. SKILL.md Protocol

### Write Flow

```text
1. Filename generation:
   - slug from title (lowercase, hyphens, ASCII only)
   - date prefix: YYYYMMDD (current date)
   - collision: check exact path `{YYYYMMDD}-{slug}.md`, then `{YYYYMMDD}-{slug}-2.md`, `-3.md`, ... until absent (avoid glob `*` which false-matches `{slug}-analysis.md`)

2. Write file:
   - frontmatter: title, created_at (current ISO timestamp), tags
   - body content + wikilinks as needed

3. Regenerate indexes (deterministic script, not LLM):
   python3 $KNOWLEDGE_BASE_PATH/scripts/generate_indexes.ts
   - on non-zero exit: read stderr, report error to callback, abort sync

4. Sync to git (at session end):
   bash $KNOWLEDGE_BASE_PATH/scripts/sync_knowledge.sh "knowledge: {title} [{tags}]"
   - on success: done
   - on exit 1 (conflict): report "SYNC_CONFLICT" to FastAPI callback, do NOT retry
   - on other failure: report stderr to callback, do NOT retry
```

> Grace period removed. Git retains full history — previous versions always recoverable via `git log`.

### `scripts/generate_indexes.ts` — Spec

Inputs: scans all `*.md` files in `$KNOWLEDGE_BASE_PATH` (excluding files in `INDEX.md`, `moc/`, `scripts/`).

For each file, parses YAML frontmatter: `title`, `created_at`, `tags`.

**Malformed frontmatter handling:**

- No frontmatter or YAML parse error → skip file, print warning to stderr (`WARN: skipping {file}: {reason}`), continue.
- `title` missing → use filename stem as title.
- `tags` missing or empty → treat as untagged (omit from all `moc/` files, include in INDEX only).
- `created_at` missing or unparseable → use file mtime as fallback for sort order; print warning.

**Script must exit non-zero if any file fails to parse** so the caller (`sync_knowledge.sh`) can detect and report the error.

Outputs (fully regenerated on every run — stale entries from deleted files are automatically removed):

- `YYYY-MM/INDEX.md` — one entry per file in that month, sorted by `created_at` descending
- `INDEX.md` (top-level) — one entry per month directory, sorted descending
- `moc/{tag}-MOC.md` — one entry per file with that tag, sorted by `created_at` descending

Entry format (same for all):

```markdown
- [{title}](./{filename}) — {one-line summary from first non-heading line} #{tags}
```

Idempotent: running twice produces the same output. Safe to run after any write.

### `scripts/sync_knowledge.sh` — Spec

```bash
#!/usr/bin/env bash
set -euo pipefail
COMMIT_MSG="${1:-knowledge: auto-sync}"
cd "$KNOWLEDGE_BASE_PATH"

git add .
git commit -m "$COMMIT_MSG" || { echo "nothing to commit"; exit 0; }

if ! git pull --rebase origin main; then
  # Conflict or rebase failure: abort and restore working tree to pre-pull state.
  git rebase --abort
  # Undo the local commit so the working tree is clean for the next session.
  git reset --soft HEAD~1
  echo "SYNC_CONFLICT: rebase failed — manual resolution required" >&2
  exit 1
fi

git push origin main
```

> On conflict: `rebase --abort` restores the working tree, `reset --soft HEAD~1` un-commits local changes (keeps them staged for next session). Agent reports failure to FastAPI callback. No LLM-driven merge. User resolves manually.
>
> Rationale: NanoClaw is the sole writer in normal operation — conflicts only occur if user manually commits from another machine. In that rare case, correctness > automation.

### Search Flow

```bash
# query only
rg --json -C 2 "{query}" $KNOWLEDGE_BASE_PATH

# tags only — matches inline YAML (`tags: [nanoclaw]`) and block-sequence (`- nanoclaw`)
rg -l "tags:.*\b{tag}\b|^\s*-\s+{tag}\b" $KNOWLEDGE_BASE_PATH

# query + tags
rg -l "tags:.*\b{tag}\b|^\s*-\s+{tag}\b" $KNOWLEDGE_BASE_PATH | xargs rg --json -C 2 "{query}"
```

> Known limitation: block-sequence pattern may match body list items. Acceptable for MVP.

## 4. FastAPI Role

### What Remains

- `KnowledgeBaseService` Python class — **read-only** (`search`, `read` methods using `rg`)
- `SearchKnowledgeTool`, `ReadNoteTool` — inject service directly into PersonaAgent (no HTTP loopback)
- Session trigger: `on_disconnect` / idle sweep → `delegate_knowledge_summary(session_id, user_id)`
- GDrive mount: **`:ro`** (FastAPI only reads)

### KnowledgeBaseService (read-only)

```python
class KnowledgeBaseService:
    def search(self, query: str, tags: list[str]) -> list[SearchResult]: ...
    def read(self, path: str) -> str: ...
```

Internally calls `rg` subprocess. No write methods.

### Session Trigger

```python
# on_disconnect in handlers.py
stm_meta = await stm_service.get_metadata(session_id)
if not stm_meta.get("knowledge_saved") and stm.human_message_count() >= MIN_TURNS_FOR_SUMMARY:
    await stm_service.set_metadata(session_id, {"knowledge_saved": True})
    await delegate_knowledge_summary(session_id, user_id)
```

STM context delivery uses a hybrid strategy based on session length:

- **Option A (inline, default)**: `human_message_count < 30` → FastAPI serializes STM messages directly into the payload.
- **Option B (host-side fetch)**: `human_message_count >= 30` → NanoClaw **host** fetches `GET /v1/stm/{session_id}/messages` before spawning the container, then injects the full context. Container agent always receives complete context regardless of option.

`MIN_TURNS_FOR_SUMMARY` is configured in `yaml_files/services/knowledge_base.yaml` (default: `3`). The 30-turn Option B threshold is also in this config file as `stm_inline_max_turns: 30`.

Minimal payload schema (Option A):

```json
{
  "task": "knowledge_summary",
  "session_id": "<str>",
  "user_id": "<str>",
  "callback_url": "http://fastapi-host:5500/v1/callback/nanoclaw/<session_id>",
  "stm_messages": [
    {"role": "human|ai", "content": "<str>"}
  ]
}
```

Option B payload omits `stm_messages` and adds `"stm_fetch_url": "http://fastapi-host:5500/v1/stm/{session_id}/messages"`. NanoClaw host resolves it before container start.

### Known Limitations

- **rclone/git concurrency**: During `git pull --rebase`, `.md` files change momentarily. rclone may attempt to sync mid-rebase, risking 0-byte uploads or lock contention. If this occurs in practice, separate the git working directory from the rclone-watched path (post-push cron copy). Deferred to post-MVP.
